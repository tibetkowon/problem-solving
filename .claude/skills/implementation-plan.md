# 구현 계획서 작성 스킬 (코드 작성 전 필수)

## 왜 먼저 계획하는가

코드를 바로 짜기 시작하면 요구사항을 잘못 이해한 채로 많은 시간을 쓸 수 있다.
계획서를 먼저 작성하면 사용자가 방향을 확인하고 수정할 기회를 얻는다.
또한 완성된 계획서는 `/docs/plans/`에 저장되어 나중에 "왜 이렇게 설계했는지" 추적할 수 있는 이력이 된다.

---

## 실행 절차

### 1단계: 요구사항 파악

기존 코드가 있으면 관련 파일을 읽어 현재 구조를 파악한다.
요청 내용에서 다음을 정리한다:
- 무엇을 만드는가 (What)
- 어떤 방식으로 만드는가 (How)
- 포함 범위와 제외 범위

### 2단계: 계획서 작성

아래 구조로 계획서를 작성하고 사용자에게 제시한다.
모든 내용은 한국어로 작성한다. 클래스명·메서드명·어노테이션 등 기술 식별자는 영어 그대로 사용한다.

```markdown
# 📋 구현 계획서: [기능명]

## 요구사항 요약
한두 문장으로 핵심 목적을 요약한다.

## 구현 범위
- **포함**: 이번 작업에서 구현할 것
- **제외**: 이번 작업에서 다루지 않을 것

## 엔티티 / DB 변경 (해당되는 경우)
새로 추가하거나 수정되는 테이블·필드를 기술한다.

## API 설계 (해당되는 경우)
| 메서드 | 경로 | 요청 | 응답 | 설명 |
|--------|------|------|------|------|

## 작업 목록 (순서대로)
| 순서 | 작업 내용 | 대상 파일/클래스 |
|------|-----------|----------------|
| 1 | ... | ... |

## 고려사항 및 주의점
설계 결정, 트레이드오프, 예외 처리, 동시성 등 중요한 판단 사항을 기술한다.

## 예상 커밋 단위
작업 목록을 기반으로 커밋을 어떻게 나눌지 미리 계획한다.
```

### 3단계: 사용자 확인 대기

계획서 제시 후 반드시 다음 문구로 확인을 요청하고 대기한다:

> **위 계획대로 진행할까요? 수정하거나 추가할 내용이 있으면 말씀해 주세요.**
> **확인해 주시면 코드 작성을 시작하겠습니다.**

사용자가 "진행해", "ㅇㅇ", "확인" 등 긍정적인 답변을 주기 전까지 코드를 작성하지 않는다.

### 4단계: 계획서 저장

사용자가 계획을 확인하면 코드 작성 전에 계획서를 파일로 저장한다.

**저장 경로:** `/docs/plans/YYYY-MM-DD-<기능명-kebab-case>.md`

예시:
- `/docs/plans/2026-03-28-chapter-api.md`
- `/docs/plans/2026-03-28-random-problem-query.md`
- `/docs/plans/2026-03-28-redis-correct-rate.md`

`/docs/plans/` 디렉토리가 없으면 먼저 생성한다.

저장 완료 후 코드 작성을 시작한다.

---

## 계획서 저장 규칙

- **파일명**: `YYYY-MM-DD-기능명.md` (오늘 날짜 기준)
- **기능명**: 간결하게 kebab-case로 (예: `random-problem-query`, `answer-submit`)
- **내용**: 사용자가 확인한 최종 계획서 전문을 그대로 저장
- **기존 파일 덮어쓰기 금지**: 같은 날 같은 기능이면 `-v2`, `-v3` 접미사 추가

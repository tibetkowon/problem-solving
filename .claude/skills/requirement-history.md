# 요구사항 히스토리 관리 스킬 (자동 적용)

## Trigger — 언제 이 스킬을 실행하는가

다음 중 하나라도 해당되면 사용자의 별도 언급 없이 **즉시 이 스킬을 적용**한다:

- 새 기능 구현 요청을 받았을 때
- 기존 기능의 개선/수정 요청을 받았을 때
- 버그 수정 요청을 받았을 때
- 리팩토링이나 성능 개선 요청을 받았을 때

**요청 처리 전** `/docs/requirements.md`에 요청 내용을 먼저 기록하고, **작업 완료 후** 상태를 업데이트한다.

---

## 실행 절차

### 1단계: /docs/ 디렉토리 및 파일 확인

- 프로젝트 루트에 `/docs/` 디렉토리 존재 여부 확인
- 없으면 생성 후 `/docs/requirements.md` 신규 작성
- 있으면 기존 파일에 항목 추가

### 2단계: 요청 접수 시 기록 (작업 시작 전)

요청을 받는 즉시 `/docs/requirements.md`에 다음 형식으로 추가한다:

```markdown
| 날짜 | 유형 | 요청 내용 | 상태 |
|------|------|-----------|------|
| YYYY-MM-DD | 기능/개선/버그/리팩토링 | 요청 내용 요약 | 진행중 |
```

**유형 분류:**
- `기능` — 새로운 기능 추가
- `개선` — 기존 기능 수정·향상
- `버그` — 오류 수정
- `리팩토링` — 코드 구조 개선
- `문서` — 문서화 작업

### 3단계: 작업 완료 후 상태 업데이트

해당 항목의 상태를 `진행중` → `완료`로 변경한다.

---

## /docs/requirements.md 파일 구조

파일이 없을 때 아래 구조로 신규 생성한다:

```markdown
# 요구사항 히스토리

> 이 문서는 프로젝트에 요청된 모든 기능, 개선사항, 버그 수정의 이력을 관리한다.

## 요구사항 목록

| 날짜 | 유형 | 요청 내용 | 상태 |
|------|------|-----------|------|
```

### 상태 값

| 상태 | 의미 |
|------|------|
| `진행중` | 현재 작업 중 |
| `완료` | 작업 완료 |
| `보류` | 우선순위 낮음 또는 결정 대기 |
| `취소` | 요청 취소됨 |

---

## 작성 규칙

- **날짜**: 오늘 날짜 (YYYY-MM-DD 형식)
- **요청 내용**: 사용자의 원문 요청을 간결하게 요약 (1~2줄)
- **기존 내용은 절대 삭제하지 않는다** — 새 항목을 테이블 하단에 추가
- **언어**: 한국어로 작성한다

---

## 완료 보고

작업 완료 후:
```
요구사항 히스토리 업데이트 완료:
- 추가: [요청 내용 요약] → 완료
```
